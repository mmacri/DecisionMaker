Module Overview

CIP-007 (System Security Management) defines technical controls to keep Bulk Electric System (BES) Cyber Systems secure. For Seattle City Light employees—operators, engineers, or support staff—the standard translates into practical steps: lock down unnecessary services, apply patches on time, prevent and detect malware, monitor events, and manage accounts responsibly.

Key CIP-007-6 Requirements

  • R1 – Ports & Services: Enable only what is needed; document exceptions.
  • R2 – Security Patch Management: Track, evaluate, and install patches on time—or document mitigating measures.
  • R3 – Malicious Code Prevention: Maintain anti-malware protection and signature updates.
  • R4 – Security Event Monitoring: Log relevant events and review them regularly.
  • R5 – Account Management: Authorize, review, and remove accounts; enforce strong authentication.
  • R6 – System Access Controls: Harden interactive and remote access (e.g., lockouts, timeouts).

Key Terms

  • BES Cyber System (BCS): Assets whose failure could impact reliable grid operation.
  • Mitigating Measure: Temporary control used when a patch cannot be applied.
  • Least Functionality: Configuring devices to run only necessary services and ports.
  • Security Event: A logged action that may indicate abnormal or risky behavior.
  • Privileged Account: An account with admin, root, or elevated roles.

Role-Specific Guidance

Operators & Dispatch

  • Verify remote-access sessions use approved gateways and multi-factor authentication.
  • Escalate repeated failed logons or unexpected reboots to Cybersecurity Operations.
  • During switching or restoration, avoid disabling security tools unless authorized.

Engineers & OT Support

  • Document service enablement with business justification and firewall/ACL protections.
  • Lead patch evaluations; record applicability, test outcomes, and mitigations.
  • Validate log forwarding, time sync, and anti-malware health after maintenance.

Field & Vendor Coordinators

  • Control portable media: scan on approved jump hosts; log use and custody.
  • Limit vendor accounts to the minimum duration; disable immediately when work ends.
  • Capture evidence (photos of cabinet port blocks, screenshots of configs) while onsite.

1. Ports, Services, and Least Functionality (R1)

Only run the services required for the job. Disable vendor defaults (e.g., Telnet, unused web consoles), and record any exceptions with justification and risk acceptance. Before enabling a new service, validate the business need, restrict it by firewall/ACL, and document who approved it.

  • DO Request changes through the standard change-management ticket and keep approval evidence.
  • DO Verify that new devices ship with unused ports closed; confirm with a quick nmap or built-in port report.
  • DON'T Enable remote services (SSH/RDP) without access controls and logging.

Evidence to Keep

  • Service/port baseline list with last review date and approver.
  • Firewall/ACL snapshots showing restrictions.
  • Change ticket numbers tied to each exception.

Field Quick Check

# Linux example netstat -tulnp | grep -v LISTEN # Windows PowerShell Get-NetTCPConnection | Where-Object {$_.State -ne "Listen"}

2. Security Patch Management (R2)

All BCS components must be assessed for vendor patches at least once every 35 days. Apply applicable patches within 35 days of availability, or document a mitigating measure if installation is deferred (e.g., compensating firewall rule, increased monitoring).

Step: Check vendor advisories
Evidence: Screenshot of advisory portal or automated report
Step: Evaluate applicability and risk
Evidence: Ticket notes showing assessment and decision
Step: Apply patch or mitigating measure
Evidence: Change record, success logs, or mitigation description

Best Practices

  • Coordinate maintenance windows early to avoid emergency changes.
  • Test patches on representative non-production devices when available.
  • Capture before/after version info for audit trails.

Timeline Cue

Day 0–5
Vendor advisory review and risk rating.
Day 5–20
Testing on lab/representative systems.
Day 20–35
Production deployment or documented mitigation.

3. Malicious Code Prevention (R3)

Keep anti-malware software active where technically feasible. Update signatures daily (or as configured by OT policy), and run scheduled scans during low-impact windows. For systems that cannot run anti-malware, apply alternative controls such as application allow lists and controlled media handling.

  • DO Confirm real-time protection is enabled after system changes.
  • DO Quarantine or isolate infected hosts and notify Cybersecurity Operations immediately.
  • DON'T Insert removable media without scanning on an approved jump host.

Evidence to Keep

  • Daily signature update logs and scan results.
  • Exception/compensating control approvals for systems without anti-malware.
  • Portable media scan logs and custody notes.

Response Tip

If malware is suspected during operations, pause non-essential activity, isolate the device (network disconnect if safe), and call Cybersecurity Operations before rebooting to preserve evidence.

4. Security Event Monitoring (R4)

BCS components must generate security event logs (logons, failed logons, admin actions, service changes). Forward logs to the designated collector when supported; otherwise, keep local logs and review them. Operators should know how to pull a quick report for investigations.

  • Ensure time synchronization so event timestamps align with SOC investigations.
  • During incident response, preserve logs before rebooting or restoring images.
  • Use correlation alerts from the SOC as triggers for local checks or escalation.

Pro tip

Bookmark the log locations for your primary systems (e.g., Windows Event Viewer custom views, syslog paths) so you can retrieve evidence quickly.

Sample Events to Review Weekly

Repeated failed logons
New local admin account creation
Service start/stop outside change window
Configuration changes on relays or HMIs

5. Account Management (R5)

Accounts must be authorized, unique to individuals, and removed when no longer needed. Shared accounts should be avoided; if unavoidable, log and monitor their use closely.

Controls

  • Review account lists at least every 15 months; remove stale or contractor accounts promptly.
  • Use privileged access only for administrative tasks; perform daily work with standard accounts.
  • Keep joiner/mover/leaver tickets linked to account changes.

Evidence

  • Account review reports with dates and sign-off.
  • Access approval records and role definitions.
  • Time-bound vendor account logs and disablement proof.

6. System Access Controls (R6)

Harden interactive and remote access. Apply strong authentication, inactivity timeouts, and failed-login lockouts. Enforce encryption for remote protocols (SSH, TLS, secure gateway) and avoid direct internet exposure.

  • Set inactivity timeouts aligned to OT policy (e.g., 15 minutes for consoles where feasible).
  • Configure account lockouts after defined failed attempts; include alerting for lockout patterns.
  • Use session recording or logging for vendor remote work; remove access promptly when tasks finish.

Configuration Reminders

  • Disable legacy protocols (e.g., SMBv1, SSLv3) unless approved.
  • Restrict jump host access to authorized personnel and keep MFA enforced.
  • Document timeout and lockout settings with screenshots after changes.

What to Do in Emergencies

If a control must be relaxed temporarily (e.g., longer timeout for restoration), obtain documented approval, set a timer to revert, and capture logs showing when the control was restored.

Implementation Checklist

  • Ports Verify unneeded ports/services are disabled and documented.
  • Patching Confirm monthly patch review was completed and results recorded.
  • Anti-Malware Check signatures and scan schedules; document exceptions.
  • Logging Validate log forwarding and local retention; review for anomalies.
  • Accounts Audit user lists, disable inactive accounts, and confirm timeout/lockout settings.
  • Evidence Store screenshots, tickets, and approvals in the compliance repository.

Scenarios & Responses

Scenario A – Emergency Patch Required

A vendor releases a critical patch for a BCS server used in switching operations. Maintenance windows are limited.

  • Open a change ticket with risk justification and planned rollback.
  • Coordinate with the control center to schedule during lowest operational impact.
  • Capture installation logs and version verification; update the patch register.
  • If timing slips, document a mitigating firewall rule until the patch is applied.
Scenario B – Suspicious USB Device

A field engineer finds a USB drive near a substation control cabinet.

  • Do not connect it to any BCS device. Bring it to an approved jump host for scanning or submit to Cybersecurity Operations.
  • Record the incident in the security ticketing system; include location and time.
  • Review local logs for any prior USB insertions on nearby devices.
Scenario C – New Remote Access Need

A vendor requests remote troubleshooting access to a relay configuration workstation.

  • Use the standard remote access gateway with multi-factor authentication; prohibit direct internet exposure.
  • Limit access duration and scope; disable the account after the task.
  • Enable session recording or logging; review for completeness after work is done.
Scenario D – Log Review Finds Anomaly

The weekly review shows repeated failed logons on a relay HMI after hours.

  • Escalate to Cybersecurity Operations with timestamps and source addresses.
  • Check physical access logs or camera footage if available.
  • Increase monitoring for that device and verify lockout thresholds are active.

Knowledge Checks

  1. Why does CIP-007 emphasize least functionality?
    Reducing open ports and services shrinks the attack surface and lowers the chance that malware or attackers can exploit unnecessary components.
  2. What should you do if a patch cannot be applied within 35 days?
    Document the reason, implement a mitigating measure (e.g., firewall rule or monitoring), and track the patch for later installation.
  3. How are suspicious log events handled?
    Forward logs to the designated collector when possible, review local logs regularly, and escalate anomalies to Cybersecurity Operations with timestamps and system details.
  4. When are shared accounts acceptable?
    Only when technically required; they must be tightly controlled, logged, and time-bound with documented approval.
  5. How often should account lists be reviewed?
    At least every 15 months, with immediate removal of stale or contractor accounts.
  6. What should you do with portable media?
    Scan on approved jump hosts before use, log the activity, and avoid direct insertion into BCS devices.
